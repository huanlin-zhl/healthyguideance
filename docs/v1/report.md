# 分析报告

- 主文档：[design.md](design.md)
- 相关：[prompts.md](prompts.md)（advice prompt）、[data-model.md](data-model.md)、[storage.md](storage.md)、[ui-flow.md](ui-flow.md)

## 1. 报告内容结构（5 段）

LLM 输出固定五段，对应 [prompts.md §4](prompts.md) 的 schema：

| 段 | 字段 | 内容 |
|---|---|---|
| 现状小结 | `summary` | 时间窗口内体重 / 体脂率 / 内脏脂肪变化与运动总量 |
| 趋势判断 | `trend` | 变好 / 变差 / 停滞，简要原因 |
| 饮食建议 | `diet_advice` | 基于备注 + 体成分趋势的方向性建议 |
| 运动建议 | `workout_advice` | 下周建议频次、类型、强度 |
| 注意事项 | `warnings` | 异常项（如肌肉量下降、内脏脂肪上升等）；无异常返回空字符串 |

每段为一段自然语言中文，**不强制列点**。

## 2. 报告 ID 与文件名

- **ID 格式**：`yyyyMMdd-HHmmss`（基于生成时间，无 hash）
- **文件路径**：`reports/<yyyy-MM>/<id>.json`
- 示例：`reports/2026-05/20260531-153000.json`

同一秒内不会生成两份报告（手动触发 + 必有 LLM 调用耗时，物理上排除冲突）。

## 3. 时间窗口

### 3.1 边界定义

- **end**：报告生成时刻（精确到秒，存入 `window.end`）
- **start**：end 往前推 N 天后，**对齐到当天 0:00:00**

举例：2026-05-31 15:30:00 触发"过去 30 天"
- end = `2026-05-31T15:30:00`
- start = `2026-05-01T00:00:00`（5 月 1 日零点）

### 3.2 预设窗口

| 预设标识 | start 推算 |
|---|---|
| `7d` | 7 天前 0:00 |
| `30d` | 30 天前 0:00 |
| `90d` | 90 天前 0:00 |
| `custom` | 用户自选日期范围（均对齐到 0:00 / 23:59:59） |

预设标识写入 `window.preset`，便于报告对比与统计。

## 4. 标题与摘要（UI 展示用）

### 4.1 标题（客户端拼装，非 LLM 生成）

格式：`<起始日期> 至 <结束日期> 分析报告`

例：`2026-05-01 至 2026-05-31 分析报告`

仅取日期部分（不含时间），美观且足够区分。

### 4.2 摘要（用于报告列表卡片）

取 `content.summary` 前 **80 字**，超出截断加 `…`。

不让 LLM 单独输出 `title` 或 `tldr` 字段——避免冗余，`summary` 第一句话天然就是核心结论。

## 5. 空数据处理

报告生成前先按窗口扫描：

| 窗口内数据情况 | 处理 |
|---|---|
| 体成分 + 运动 + 备注 三者都为 0 条 | **不调 LLM**，直接弹错误对话框："该时间窗口内无数据，请先导入截图或调整窗口" |
| 体成分 0 条但有运动或备注 | 仍调用 LLM；prompt 中数据段为空数组，LLM 在 `summary`/`trend` 中明确说明"数据不足" |
| 仅有体成分 1 条 | 同上；LLM 应给出"数据太少，无法判断趋势" |

## 6. report.json 落盘结构

```jsonc
{
  "id":          "20260531-153000",
  "generated_at": "2026-05-31T15:30:00",
  "window": {
    "start":  "2026-05-01T00:00:00",
    "end":    "2026-05-31T15:30:00",
    "preset": "30d"
  },
  "goal": {
    "weight_kg":    78,
    "body_fat_pct": 20
  },
  "source": {
    "model":            "gpt-4o",
    "api_version":      "2024-10-21",
    "workout_ids":      ["20260527-193500_workout_b2c3d4e5", ...],
    "body_metrics_ids": ["20260529-072300_body-metrics_e5f6g7h8", ...],
    "notes_window":     ["2026-04.txt", "2026-05.txt"]
  },
  "content": {
    "summary":        "...",
    "trend":          "...",
    "diet_advice":    "...",
    "workout_advice": "...",
    "warnings":       ""
  },
  "disclaimer": "本建议仅供参考，不构成医疗或营养专业意见。"
}
```

### 6.1 字段说明

| 字段 | 来源 | 说明 |
|---|---|---|
| `id` | 客户端 | 见 §2 |
| `generated_at` | 客户端 | 报告写盘时刻 |
| `window` | 客户端 | 见 §3 |
| `goal` | 客户端 | 从 settings 读取，**写入快照**；将来用户改了目标，老报告仍保留当时值 |
| `source.model` / `api_version` | 客户端 | 调 LLM 时使用的版本 |
| `source.workout_ids` / `body_metrics_ids` | 客户端 | 喂给 LLM 的记录 id 列表，便于追溯 |
| `source.notes_window` | 客户端 | 喂给 LLM 的 notes 文件名列表 |
| `content` | LLM | 五段输出 |
| `disclaimer` | 客户端 | 固定文案，不让 LLM 生成 |

### 6.2 不存的字段

- **不存** `title` —— 列表展示时按 §4.1 实时拼
- **不存** `tldr` / `excerpt` —— 同 §4.2
- **不存** `schema_version` —— 与 [data-model.md §10](data-model.md) 保持一致

## 7. 报告列表的展示策略

- 按 `generated_at` 倒序排列
- 每张卡片显示：
  - 标题（§4.1）
  - 摘要（§4.2）
  - 生成时间（相对时间，如"3 天前"）
  - `preset` 标签（`30d` 等）
- 点卡片进详情页，详情页按五段渲染

## 8. 报告详情页渲染

- 顶部：标题 + 窗口范围 + 生成时间 + `preset` 标签
- 主体：5 段卡片化展示，每段带标题图标
  - 📊 现状小结
  - 📈 趋势判断
  - 🥗 饮食建议
  - 💪 运动建议
  - ⚠ 注意事项（`warnings` 为空时不显示这块）
- 底部：免责声明（小字灰色）
- 右上角操作：`删除`、`重新生成`（按同样窗口再生成一份）

## 9. 缓存与重算

- 报告生成后**永久保留**，不自动清理
- "重新生成" = 按同样窗口再调一次 LLM，**生成新 ID 的新报告**，旧报告保留
- 报告列表无上限（实际不会爆）

## 10. 不在范围

- 报告对比（P1）
- 报告导出 PDF / Markdown（P2）
- 自动生成 / 定时生成（已明确不做）
